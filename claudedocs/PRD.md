# PRD: inflearn-todo

## 1. 개요

### 프로젝트 목적
팀 구성원이 할 일을 함께 등록, 배정, 관리하며 마감일과 카테고리 기준으로 업무 진행 상황을 추적할 수 있는 협업형 할 일 관리 도구를 제공한다.

### 타겟 사용자
- 3~10명 규모의 소규모 팀 또는 프로젝트 그룹 구성원
- 여러 명이 할 일을 함께 확인하고, 특정 인원에게 할 일을 배정하며, 진행 상황을 공유해야 하는 사용자
- 별도의 서버 인프라 구축 없이 로컬 환경에서 팀 업무를 관리하고자 하는 사용자

### 핵심 가치 제안
팀원 모두가 동일한 할 일 목록을 공유하며, 마감일 기반 알림과 카테고리 분류를 통해 "무엇을, 언제까지, 어떤 기준으로" 처리해야 하는지 한눈에 파악할 수 있게 한다.

## 2. 기능 요구사항

### 기능 1: 할 일 추가/수정/완료/삭제 (기본 CRUD)

**사용자 스토리**
> 팀원으로서, 할 일을 새로 등록하고 필요할 때 내용을 수정하거나 완료 처리 또는 삭제하고 싶다, 팀의 작업 목록을 항상 최신 상태로 유지하기 위해

**수용 기준**
- [ ] 사용자가 제목(1자 이상 100자 이하)을 입력하여 할 일을 등록하면, 등록된 할 일이 목록 최상단에 즉시 표시된다.
- [ ] 사용자가 할 일 항목의 완료 버튼을 선택하면, 해당 항목은 "완료" 상태로 표시되고 목록 내 미완료 항목과 시각적으로 구분되며, 완료 처리 순간 시각적 피드백(모션)과 함께 "완료되었습니다" 안내가 잠시 나타난다.
- [ ] 사용자가 할 일 항목의 수정 버튼을 선택하면 별도의 새 창(팝업)을 띄우지 않고 목록 내 해당 항목이 그 자리에서 바로 제목/마감일/카테고리를 고칠 수 있는 입력 형태로 바뀌며, 기존 값이 채워져 있다.
- [ ] 수정 입력 중 Enter를 누르면 즉시 수정이 완료되어 저장되고 목록에 변경 내용이 반영된다.
- [ ] 수정 시에도 등록과 동일한 검증 규칙(제목 1~100자, 마감일 오늘 이후, 카테고리 0~5개)이 적용되어, 규칙을 위반하면 등록 때와 동일한 안내 문구와 함께 저장이 차단된다.
- [ ] 사용자가 할 일 항목을 삭제하면, 확인 절차(삭제 확인 대화상자) 없이는 삭제가 진행되지 않으며 삭제 확정 시 목록에서 즉시 제거된다.
- [ ] 제목을 입력하지 않고 등록을 시도하면 "제목을 입력해주세요"라는 안내 문구와 함께 등록이 차단된다.
- [ ] 이미 삭제된 할 일 항목에 대해 완료 처리, 수정 또는 삭제를 시도하면(예: 다른 팀원이 먼저 삭제한 경우) "존재하지 않는 항목입니다" 오류가 표시되고, 삭제 확인 대화상자는 닫히지 않고 오류를 보여준 채로 유지된다.

### 기능 2: 마감일 & 알림

**사용자 스토리**
> 팀원으로서, 할 일에 마감일을 설정하고 마감이 임박하면 알림을 받고 싶다, 기한 내에 작업을 놓치지 않고 완료하기 위해

**수용 기준**
- [ ] 사용자가 할 일 등록 또는 수정 시 마감일(연-월-일, 필요 시 시:분)을 지정하면, 해당 값이 할 일 항목에 저장되어 목록에 표시된다.
- [ ] 마감일이 오늘 날짜를 기준으로 24시간 이내로 남은 미완료 할 일은 목록에서 "임박" 표시(예: 별도 색상 또는 라벨)와 함께 상단 알림 영역에 노출된다.
- [ ] 마감일이 현재 시각보다 이전이고 아직 완료되지 않은 할 일은 "기한 초과"로 표시된다.
- [ ] 마감일을 과거 날짜로 설정하려는 경우 "마감일은 오늘 이후로 설정해주세요" 경고 문구가 표시되고 저장이 차단된다.
- [ ] 마감일을 설정하지 않고 저장하면 해당 할 일은 "마감일 없음" 상태로 정상 등록되며 알림 대상에서 제외된다.
- [ ] 할 일 등록 시 마감일을 설정하고자 하면, 별도 입력 없이도 현재 날짜/시각이 기본값으로 채워져 사용자는 이후 시점으로 조정만 하면 되도록 편의를 제공한다.

### 기능 3: 카테고리 분류 (관리형 카테고리 선택)

**사용자 스토리**
> 팀원으로서, 별도로 관리되는 카테고리 목록 중에서 골라 할 일에 지정하고 싶다, 담당 업무 영역이나 프로젝트별로 할 일을 일관된 기준으로 구분해서 확인하기 위해

**수용 기준**
- [ ] 카테고리는 자유 입력(태그 직접 타이핑) 방식이 아니라, 별도의 카테고리 관리 화면에서 등록/삭제하여 관리하는 목록 중에서 선택하는 방식으로 동작한다.
- [ ] 카테고리 관리 화면에서 새 카테고리 이름(1자 이상 30자 이하)을 등록하면, 해당 카테고리가 즉시 목록에 추가되어 할 일 등록/수정 시 선택 가능한 항목으로 나타난다.
- [ ] 이미 존재하는 이름(대소문자 무관)으로 카테고리를 등록하려 하면 "이미 존재하는 카테고리입니다" 안내와 함께 등록이 차단된다.
- [ ] 카테고리 이름을 입력하지 않고 등록을 시도하면 "카테고리 이름을 입력해주세요" 안내와 함께 등록이 차단된다.
- [ ] 할 일을 등록하거나 수정할 때는 관리 중인 카테고리 목록 중 드롭다운으로 최대 1개까지만 간단히 선택할 수 있다(선택하지 않아도 됨).
- [ ] 카테고리를 하나도 선택하지 않고 저장하면 해당 할 일은 "미분류"로 자동 분류되어 목록에 표시된다.
- [ ] 사용자가 특정 카테고리를 선택하여 목록을 필터링하면, 해당 카테고리가 지정된 할 일만 화면에 표시된다.
- [ ] 카테고리 이름을 수정하면, 이미 그 카테고리가 지정되어 있던 기존 할 일들도 표시값이 새 이름으로 함께 갱신된다.
- [ ] 카테고리 이름을 다른 카테고리와 중복되는 이름으로 수정하려 하면 "이미 존재하는 카테고리입니다" 안내와 함께 수정이 차단된다.
- [ ] 사용자가 카테고리 목록의 순서를 드래그 앤 드롭으로 자유롭게 변경할 수 있고, 변경된 순서는 이후에도 계속 유지된다.
- [ ] 카테고리를 삭제하면 이후 할 일 등록/수정 시 선택 목록에서 제거되며, 이미 그 카테고리가 지정되어 있던 기존 할 일은 표시값이 그대로 유지된다(삭제는 소급 변경하지 않음).

## 3. 비기능 요구사항

### 성능
- 할 일 목록이 100건 이하일 때, 항목 추가/완료/삭제/필터링 조작 후 화면 갱신까지 걸리는 시간은 1초 이내여야 한다.
- 마감일 임박 알림은 마감 24시간 전 시점 기준 5분 이내의 오차 범위 내에서 노출되어야 한다.

### 보안
- 팀 내 공유를 전제로 하되, 파일 접근 권한은 로컬 파일 시스템의 기본 접근 제어(운영체제 사용자 권한)를 따른다.
- 할 일 데이터에는 비밀번호, 결제 정보 등 민감 정보를 포함하지 않는다.

### 접근성
- 화면 너비 375px 이상 환경에서 주요 기능(추가, 완료, 삭제, 필터링)이 깨짐 없이 동작해야 한다.
- 완료/미완료, 임박/기한초과 상태는 색상뿐 아니라 텍스트 라벨로도 구분되어야 한다(색각 이상 사용자 고려).
- 모바일 폭(`md` 브레이크포인트 미만)에서는 카테고리 수정/삭제 아이콘과 순서 변경(드래그 앤 드롭) 기능은 노출되지 않으며, 카테고리 선택(필터링) 기능만 제공된다.

## 4. 제약사항

### 기술적 제약
- 별도의 서버나 데이터베이스 없이, 로컬 환경의 JSON 파일을 직접 읽고 쓰는 방식(File-based JSON)으로 할 일 데이터를 저장한다.
- **트레이드오프 명시**: "팀/협업용"이라는 타겟과 "서버 없는 로컬 File-based JSON 저장"이라는 요구사항은 근본적으로 긴장 관계에 있다.
  - 여러 팀원이 동시에 동일한 JSON 파일에 접근하여 수정할 경우, 마지막에 저장한 내용이 이전 내용을 덮어쓰는 동시 편집 충돌이 발생할 수 있다.
  - 실시간 동기화(한 사용자의 변경 사항이 다른 사용자 화면에 즉시 반영되는 기능)는 지원하지 않는다. 팀원 간 최신 데이터 공유는 파일을 공유 저장소(예: 공유 폴더)에 두고 각자 새로고침하는 방식에 의존한다.
  - 여러 팀원이 이 JSON 파일에 실제로 어떻게 접근하고 공유할 것인지(공유 폴더 경로, 파일 잠금 여부, 백업 정책 등)는 본 PRD의 범위가 아니며 TECH_SPEC 단계에서 구체적으로 결정한다.
- 오프라인(네트워크 미연결) 환경에서도 로컬 파일 기반으로 정상 동작해야 한다.
- **클라우드(서버리스) 배포 미지원**: 현재의 File-based JSON 저장 방식은 로컬 상시 실행(`next dev`/`next start`) 환경을 전제로 설계되었다. Vercel과 같은 서버리스 플랫폼은 함수 실행마다 파일시스템이 격리·재시작되어(쓰기 가능 영역이 `/tmp`뿐) 이 저장 방식과 근본적으로 호환되지 않으며, 배포 시 조회는 가능하나 쓰기(추가/수정/삭제)는 대부분 실패하거나 데이터가 유지되지 않는다. 이는 알려진 한계로 남겨두며, 실제 클라우드 프로덕션 운영이 필요해지면 저장 계층을 영속 저장소(DB 등)로 마이그레이션하는 별도 작업이 필요하다.

### 비즈니스 제약
- MVP 범위는 위 3개 기능으로 한정하며, 사용자 인증/권한 관리, 실시간 협업, 알림의 외부 채널(이메일, 메신저) 연동은 포함하지 않는다.
- 별도의 예산 또는 일정 제약은 명시되지 않았으며, MVP 완성 후 확장 여부를 재논의한다.

## 5. 성공 지표
- 팀원 1명이 신규 할 일을 등록하고 마감일과 카테고리를 지정하는 전체 과정을 90초 이내에 완료할 수 있다.
- 마감이 임박(24시간 이내)한 미완료 할 일이 목록 상단 알림 영역에 100% 노출된다.
- 카테고리 필터링 시 필터 조건에 맞는 할 일만 정확히 표시되는 비율이 100%이다(오탐/누락 0건).
